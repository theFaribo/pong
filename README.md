Pong SOLID in C
Учебный проект с консольной игрой Pong на языке C.
Проект сделан не как полноценная игра, а как демонстрационный пример для объяснения принципов SOLID в языке C.
Важно: код написан в учебных целях и на момент публикации не запускался и не проверялся полностью. Не стоит копировать его вслепую. Перед использованием нужно самостоятельно собрать проект, проверить компиляцию, протестировать поведение игры и исправить возможные ошибки.
Цель проекта
Показать, что SOLID применим не только в объектно-ориентированных языках, но и в C.
В C нет классов, наследования и интерфейсов в привычном виде, но есть те же архитектурные проблемы:
* слишком большие функции;
* смешивание игровой логики, ввода и вывода;
* сложность тестирования;
* жёсткая привязка к getchar() и printf();
* трудность изменения отдельных частей программы.
Этот проект показывает, как можно разделить код по зонам ответственности.
Важное замечание
Этот проект не является примером идеальной архитектуры.
Для маленькой консольной игры такое разделение может быть избыточным. Здесь оно сделано специально, чтобы на простом коде показать, как принципы SOLID можно применять в C.
Главная идея проекта — не сделать идеальный Pong, а показать подход:
input отдельно
game logic отдельно
physics отдельно
render отдельно
main только связывает модули

Требования
Для сборки нужен компилятор C, например gcc.
Проект рассчитан на стандарт C11.
Как собрать проект
Из корня проекта выполнить:
gcc -Wall -Wextra -Werror -std=c11 src/*.c -o pong
Если компиляция прошла успешно, появится исполняемый файл:
pong
На macOS/Linux запуск:
./pong
Как играть
Управление:
A / Z — движение левой ракетки вверх / вниз
K / M — движение правой ракетки вверх / вниз
Space — пропуск хода
Игра заканчивается, когда один из игроков набирает 21 очко.

Что демонстрирует проект
S — Single Responsibility Principle
Каждый модуль отвечает за свою область:
* input.c — чтение и разбор пользовательского ввода;
* game.c — состояние игры, счёт, игровой цикл обновления;
* physics.c — движение мяча, столкновения, проверки попаданий;
* render.c — отрисовка игрового поля в консоль;
* main.c — точка входа и связывание модулей.
O — Open/Closed Principle
Команды пользователя сначала преобразуются в абстрактное действие Action.
Это позволяет менять клавиши управления в input.c, не переписывая игровую логику.
L — Liskov Substitution Principle
В C этот принцип можно понимать через заменяемые контракты.
Например, источник команд можно заменить:
* с консольного ввода;
* на тестовый ввод;
* на бота;
* на сетевой ввод.
Главное — чтобы новый источник команд возвращал корректные значения типа Action.
I — Interface Segregation Principle
Код разделён на небольшие .h файлы.
Каждый модуль подключает только те объявления, которые ему действительно нужны.
D — Dependency Inversion Principle
Игровая логика не должна напрямую зависеть от деталей ввода и вывода.
game.c работает с состоянием игры и действием Action, а не с getchar() или printf().

Возможные проблемы
Так как код учебный и не был полноценно проверен, возможны ошибки:
* проблемы компиляции;
* неточное поведение мяча;
* некорректная обработка ввода;
* особенности отображения в разных терминалах;
* необходимость дополнительной очистки экрана вместо печати пустых строк.
Перед использованием код нужно проверить самостоятельно.
Что можно улучшить
Возможные направления развития:
* добавить Makefile;
* добавить автоматические тесты для game.c и physics.c;
* заменить печать пустых строк нормальной очисткой терминала;
* добавить replay-тесты с заранее заданной последовательностью команд;
* отделить правила игры от конкретных размеров поля;
* добавить конфигурацию клавиш управления;
* сделать тестовый источник команд без getchar();
* вынести paddle-логику в отдельный модуль;
* добавить CI-сборку с gcc -Wall -Wextra -Werror.
